## 一、项目背景

### 1.1 NoCodeFlow 是什么？

NoCodeFlow 是一个 **AI 辅助零代码开发平台**，定位为 **AI 认知工作台**。用户通过自然语言与 AI Agent（基于 Claude CLI）交互，完成软件开发、文档分析、知识管理等任务。

### 1.2 解决什么问题？

- **降低开发门槛**：非技术人员也能通过对话构建应用
- **统一工作空间**：将代码编辑、AI 对话、文件管理整合到一个界面
- **安全可控**：通过权限沙箱确保 AI 操作不会越权

### 1.3 整体架构

```
用户界面 (React + shadcn/ui)
    ↓
IPC 通信层 (Electron IPC)
    ↓
主进程 (Node.js + Electron)
    ↓
文件系统 / 配置存储 / Agent Runtime
```

### 4.2 核心模块职责

#### 主进程 (`electron/main/`)

- **index.ts**：应用生命周期管理，注册 IPC 处理程序
- **window.ts**：创建和管理 BrowserWindow
- **ipc/**：处理渲染进程的请求，操作文件系统、配置、Agent
- **agent/**：管理 Claude CLI 进程，解析输出流
- **store/**：electron-store 实例，持久化配置

#### 渲染进程 (`src/`)

- **components/**：UI 组件，按功能划分
- **stores/**：Zustand 状态管理，处理业务逻辑
- **hooks/**：自定义 Hooks，封装 IPC 调用
- **lib/**：工具函数，事件总线，解析器

#### 共享代码 (`shared/`)

- **types/**：类型定义，确保主进程和渲染进程类型一致
- **constants/**：全局常量，如最大文件深度、超时时间等

## 五、关键逻辑说明

### 5.1 IPC 通信机制

**问题**：Electron 的主进程和渲染进程是隔离的，如何安全通信？

**解决方案**：

1. **preload 脚本**：通过 `contextBridge` 暴露有限的 API
   ```typescript
   // electron/preload/index.ts
   contextBridge.exposeInMainWorld('api', {
     invoke: (channel: string, ...args: unknown[]) => ipcRenderer.invoke(channel, ...args),
     on: (channel: string, callback: (...args: unknown[]) => void) => { ... }
   });
   ```

2. **频道常量**：`shared/types/ipc.ts` 定义所有频道，防止硬编码
   ```typescript
   export const IPC_CHANNELS = {
     FS_READ: 'fs:read',
     FS_WRITE: 'fs:write',
     CONFIG_GET: 'config:get',
     // ...
   } as const;
   ```

3. **类型安全**：TypeScript 编译时检查频道和参数类型

**踩坑点**：
- 必须在 preload 中验证频道是否在允许列表中
- 渲染进程不能直接调用 `ipcRenderer`，必须通过 preload 暴露的 API

### 5.2 配置存储与加密

**问题**：Claude API Key 如何安全存储？

**解决方案**：

1. **electron-store**：将配置持久化到用户目录
2. **简单加密**：API Key 使用 XOR 加密，防止明文泄露
   ```typescript
   // electron/main/store/config.ts
   function encrypt(text: string, key: string): string {
     // 简单的 XOR 加密，不是安全加密，只是防止明文存储
     // ...
   }
   ```

3. **读取时解密**：从存储读取时自动解密

**踩坑点**：
- electron-store 的数据存储在用户目录，不是应用目录
- 配置变更后需要通知渲染进程更新

### 5.3 文件树构建

**问题**：如何高效构建文件树，避免扫描 `node_modules` 等大目录？

**解决方案**：

1. **忽略列表**：`IGNORED_DIRS` 包含常见大目录
   ```typescript
   const IGNORED_DIRS = new Set([
     'node_modules', '.git', 'dist', '.next', '__pycache__'
   ]);
   ```

2. **深度限制**：`FILE_TREE_MAX_DEPTH` 限制扫描深度（默认 10 层）

3. **二进制文件检测**：读取文件头部，检查是否包含空字节

4. **文件大小限制**：超过 `FILE_MAX_SIZE_BYTES` 的文件标记为二进制

**踩坑点**：
- 符号链接可能导致无限递归，需要检测
- 权限不足的目录需要跳过，不能抛出异常
- Windows 和 macOS 的路径分隔符不同，需要统一处理

### 5.4 Agent 运行时集成

**问题**：如何集成 Claude CLI，处理流式输出？

**解决方案**：

1. **子进程管理**：使用 `child_process.spawn` 启动 Claude CLI
   ```typescript
   const proc = spawn('claude', ['-p', '--output-format', 'stream-json', message]);
   ```

2. **流式解析**：逐行解析 JSON 输出，识别事件类型
   ```typescript
   // 事件类型：text, tool_use, tool_result, error, result, system
   ```

3. **会话恢复**：通过 `session_id` 恢复会话，支持多轮对话

4. **超时与终止**：设置超时时间，超时后强制终止进程

**踩坑点**：
- Claude CLI 必须全局安装：`npm install -g @anthropic-ai/claude-code`
- Windows 上命令是 `claude.cmd`，不是 `claude`
- 流式输出可能包含不完整的 JSON，需要缓冲处理
- 进程终止需要发送 SIGTERM，等待后再 SIGKILL

### 5.5 状态管理设计

**问题**：如何管理复杂的应用状态，确保类型安全？

**解决方案**：

1. **Zustand**：轻量级状态管理，支持 TypeScript
   ```typescript
   interface WorkspaceState {
     rootPath: string | null;
     fileTree: FileNode[];
     openTabs: TabInfo[];
     // ...
   }
   export const useWorkspaceStore = create<WorkspaceState>((set, get) => ({ ... }));
   ```

2. **IPC 调用封装**：统一的 `ipcInvoke` 函数，处理错误
   ```typescript
   async function ipcInvoke<T>(channel: string, ...args: unknown[]): Promise<T> {
     const result = await window.api.invoke(channel, ...args);
     if (result.error) throw new Error(result.error);
     return result.data as T;
   }
   ```

3. **脏标记**：编辑器内容变更后标记为 dirty，保存后清除

**踩坑点**：
- Zustand 的 `set` 函数是浅合并，嵌套对象需要展开
- 异步操作需要处理 loading 和 error 状态
- 多个组件共享状态时，注意避免不必要的重渲染

### 5.6 主题系统

**问题**：如何实现浅色/深色/跟随系统主题？

**解决方案**：

1. **CSS 变量**：`src/index.css` 定义主题变量
   ```css
   :root { --background: 0 0% 100%; }
   .dark { --background: 222.2 84% 4.9%; }
   ```

2. **Tailwind dark 模式**：使用 `dark:` 前缀
   ```tsx
   <div className="bg-background dark:bg-foreground">
   ```

3. **运行时切换**：通过 `document.documentElement.classList` 切换主题

**踩坑点**：
- 系统主题需要监听 `prefers-color-scheme` 变化
- 主题变更需要持久化到配置
- 部分组件可能需要强制重渲染

## 七、常见问题

### Q1: 应用启动后白屏？

**可能原因**：
- Vite 开发服务器未启动
- preload 脚本路径错误
- TypeScript 编译错误

**解决方案**：
1. 检查终端是否有错误信息
2. 确认 `npm run electron:dev` 正在运行
3. 检查 `electron/preload/index.ts` 是否正确导出

### Q2: Agent 无法启动？

**可能原因**：
- Claude CLI 未安装
- API Key 配置错误
- 网络连接问题

**解决方案**：
1. 终端运行 `claude --version` 检查安装
2. 在设置页面检查 Claude 配置
3. 检查 `ANTHROPIC_BASE_URL` 和 `ANTHROPIC_API_KEY` 环境变量

### Q3: 文件树不显示？

**可能原因**：
- 工作区路径无效
- 权限不足
- 忽略规则误匹配

**解决方案**：
1. 检查 `rootPath` 是否正确
2. 确认有目录读取权限
3. 检查 `IGNORED_DIRS` 是否包含目标目录

### Q4: 编辑器无法保存？

**可能原因**：
- 文件被其他程序占用
- 磁盘空间不足
- 权限问题

**解决方案**：
1. 检查文件是否被锁定
2. 确认磁盘空间
3. 检查文件写入权限
